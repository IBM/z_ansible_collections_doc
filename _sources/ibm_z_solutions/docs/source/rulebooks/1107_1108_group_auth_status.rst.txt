1107_1108_group_auth_status - Monitor zSecure Alerts from Kafka for Group Authority Change
=====================================

.. contents::
   :local:
   :depth: 1



Synopsis
--------

This rulebook monitors RACF group authority change events generated by zSecure and delivered through Kafka. It triggers a response whenever group authority is granted (alert C2P1107I) or removed (alert C2P1108I) for a RACF user.

When either alert is received, the rulebook launches the configured AAP job template to perform the response workflow.

Unlike rulebooks that use multi event correlation, this rulebook uses a single event condition. Each zSecure alert in this family is self-contained, so no corroborating event is required and no timeout is needed.

Rulebook
--------

.. code-block:: yaml

   - name: Monitor zSecure Alerts from Kafka for Group Authority Change
     hosts: all
     sources:
       - name: kafka
         ansible.eda.kafka:
           topic: "{{ kafka_topic }}"
           host: "{{ kafka_host }}"
           port: "{{ kafka_port }}"
           security_protocol: "{{ security_protocol }}"
           ssl_cafile: "{{ cafile }}"

         filters:
           - ibm.ibm_eda_zos.security_alerts:
               event_source: "kafka"
     rules:
       - name: Handle Group Authority Granted or Removed (C2P1107I, C2P1108I)
         condition: >
           event.body.alert_code == "C2P1107I" or event.body.alert_code == "C2P1108I"
         action:
           run_job_template:
             name: zSecure - Respond to Group Authority Change
             organization: "Default"

Parameters
---------

Sources
~~~~~~~

**kafka**

Connects to a Kafka broker to consume zSecure alert messages.

**topic**
   The Kafka topic name that carries zSecure alerts.
   
   :required: True
   :type: str

**host**
   The Kafka broker hostname or IP address.
   
   :required: True
   :type: str

**port**
   The Kafka broker port number.
   
   :required: True
   :type: int

**security_protocol**
   The security protocol for the Kafka connection. Common values are SSL and PLAINTEXT.
   
   :required: True
   :type: str

**ssl_cafile**
   Path to the CA certificate file used for SSL/TLS verification.
   
   :required: True (when using SSL)
   :type: str

Filters
-------

**ibm.ibm_eda_zos.security_alerts**

Filter plugin that parses and structures zSecure alert messages from Kafka events. Without this filter the rulebook condition will not match because the alert code field will not exist in the raw payload.

**event_source**
   Specifies the source type of the event stream.
   
   :required: True
   :type: str
   :choices: kafka

Rules
-----

**Handle Group Authority Granted or Removed**

Alert codes monitored:

* **C2P1107I**: zSecure alert indicating group authority granted to a RACF user.
* **C2P1108I**: zSecure alert indicating group authority removed from a RACF user.

Condition
^^^^^^^^^

This rule uses a single event condition that matches either alert code:

.. code-block:: python

   event.body.alert_code == "C2P1107I" or event.body.alert_code == "C2P1108I"

The rule fires immediately when any one incoming event satisfies the condition. There is no correlation window because each zSecure alert is self-contained and no second event is required.

Why both alert codes share one rule
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Both alert codes represent a mutation to RACF group membership, one additive and one subtractive. The security response is the same in both cases, so combining them in one rule keeps both events on the same audit trail.

If your security policy requires different responses for granted versus removed authority, this single rule can be split into two rules, each targeting one alert code with its own action.

Action
^^^^^^

Launches the AAP job template zSecure - Respond to Group Authority Change in the Default organization. The matched event is passed to the job template through ansible_eda.event. The response workflow performed by the job template is documented on the corresponding playbook page in this collection.

Event structure
---------------

Event (C2P1107I or C2P1108I)
~~~~~~~~~~~~~~~~~~~~~~~~~

.. code-block:: json

   {
     "body": {
       "alert_code": "C2P1107I",
       "alert_message": "GROUP AUTHORITY GRANTED TO USER123 FOR GROUP SYSPROG",
       "hostname": "ZSYS01",
       "target_user": "USER123",
       "group_name": "SYSPROG",
       "timestamp": "2024-01-15T10:30:00Z"
     },
     "meta": {
       "received_at": "2024-01-15T10:30:01Z"
     }
   }

Event body fields
-----------------

* **alert_code**: the zSecure alert code (C2P1107I or C2P1108I).
* **alert_message**: descriptive message about the group authority change.
* **hostname**: the z/OS system where the event occurred.
* **target_user**: the RACF user ID affected by the change.
* **group_name**: the RACF group name involved in the change.
* **timestamp**: ISO 8601 timestamp of the alert.

Variables
---------

When you activate the rulebook in Ansible Automation Platform, the following variables are defined:

.. code-block:: yaml

   kafka_topic: "zsecure-alerts"
   kafka_host: "kafka.example.com"
   kafka_port: 9093
   security_protocol: "SSL"
   cafile: "/path/to/ca-cert.pem"

Examples
---------

Example 1: Basic Activation
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Create a rulebook activation in Ansible Automation Platform with the following activation variables:

.. code-block:: yaml

   kafka_topic: "zsecure-security-alerts"
   kafka_host: "kafka-broker.company.com"
   kafka_port: 9093
   security_protocol: "SSL"
   cafile: "/etc/kafka/certs/ca-cert.pem"

Example 2: Testing the Rule
~~~~~~~~~~~~~~~~~~~~~~~~~~

To test the rulebook, publish a sample event to your Kafka topic.

Publish a C2P1107I event:

.. code-block:: bash

   echo '{
     "body": {
       "alert_code": "C2P1107I",
       "alert_message": "GROUP AUTHORITY GRANTED TO TESTUSER FOR GROUP SYSPROG",
       "hostname": "ZSYS01",
       "target_user": "TESTUSER",
       "group_name": "SYSPROG",
       "timestamp": "2024-01-15T10:30:00Z"
     },
     "meta": { "received_at": "2024-01-15T10:30:01Z" }
   }' | kafka-console-producer \
        --broker-list kafka-broker:9093 \
        --topic zsecure-alerts

Replace the alert code value with C2P1108I to test the authority removed path.

Example 3: Splitting Granted and Removed into Two Rules
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

If your policy requires different actions for granted versus removed authority, split the single rule into two:

.. code-block:: yaml

   rules:
     - name: Handle Group Authority Granted (C2P1107I)
       condition: event.body.alert_code == "C2P1107I"
       action:
         run_job_template:
           name: zSecure - Respond to Group Authority Granted
           organization: "Default"
    
     - name: Handle Group Authority Removed (C2P1108I)
       condition: event.body.alert_code == "C2P1108I"
       action:
         run_job_template:
           name: zSecure - Respond to Group Authority Removed
           organization: "Default"

Notes
-----

* The rulebook runs continuously, monitoring the Kafka topic for new events.
* Each event is processed independently. There is no correlation window or timeout.
* Ensure that you have installed the zsecure filter plugin in the decision environment for the condition to evaluate correctly.
* Before you activate the rulebook, ensure that the referenced AAP work template exists.
* System clocks should be synchronised between Kafka, AAP and z/OS for accurate event timestamps.

Troubleshooting
---------------

Rulebook not triggering
~~~~~~~~~~~~~~~~~~~~~~~

* Verify whether the C2P1103I events are being published to Kafka.
* Verify whether the event format matches the expected structure.
* Verify whether the alert code field is present and exactly matches.
* Review activation logs for errors.

Events not matching
~~~~~~~~~~~~~~~~~~

* Enable verbose logging in activation settings.
* Verify whether the alert_code field value is exact (case sensitive).
* Check that the filter plugin is parsing events correctly.
